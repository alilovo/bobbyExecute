# Forensics Update Map v1 — High-Detailed Spec

## Status

Draft for architecture alignment, implementation planning, and workflow integration.

## Purpose

Dieses Dokument definiert die Zielarchitektur, die Datenverträge, die Pipeline-Phasen, die Handoffs, die Scoring-Logik und die Implementierungsreihenfolge für eine erweiterte **Onchain Forensics Engine**.

Die Forensics Engine soll **kein eigener autoritativer Strategy-Bot** sein, sondern eine **Shared Onchain Forensics Evidence Plane**, die drei Workflows strukturiert versorgt:

1. **Meta Fetch Engine** — strategischer Layer
2. **Low Cap Hunter** — opportunistischer Layer
3. **Shadow Intelligence** — Monitoring-Layer

---

# 1. Executive Summary

## 1.1 Core Positioning

Die bestehende Codebasis ist bereits stark in:

* launch detection
* deployer risk
* bundler / coordination fingerprints
* token-extension / authority risk
* migration tracking
* outcome labeling
* post-rug tracing

Sie ist aber noch nicht vollständig als allgemeine Onchain Intelligence Plane verdrahtet.

Die stärkste Zielarchitektur ist deshalb:

* **Meta / X / Narrative Layer**
* **Onchain Forensics Evidence Plane**
* **Workflow Orchestration**

  * Meta Fetch Engine
  * Low Cap Hunter
  * Shadow Intelligence
* **Deterministic Core** als einzige autoritative Decision-/Execution-Schicht

## 1.2 Main Upgrade Goal

Die Forensics-Codebasis soll von einer überwiegend launch-/rug-orientierten Analyse in eine **shared, replayable, contract-first evidence plane** überführt werden.

## 1.3 Hard Constraints

* keine LLM-Authority
* keine MCP-Authority
* keine Sidecar-Authority
* kein zweiter Decision-Truth-Pfad
* fail-closed
* journal-first
* replay-capable
* provenance-required
* deterministic core remains authority shell

# 10. Scoring and Verdict Logic

## 10.1 Forensic Dimension Set

* deployer_risk
* bundler_toxicity
* contract_risk
* holder_distribution
* liquidity_integrity
* attention_quality
* state_change_quality

## 10.2 Hard Reject Conditions

A token should hard reject if any of the following cross configured fail thresholds:

* extreme bundler dominance
* same-slot bundle at hard-fail level
* unsafe freeze/mint authority combination
* withdrawable LP risk without mitigation
* severe deployer-linked holder concentration
* confirmed recovery sweep after known failure outcome

## 10.3 Soft Watch Conditions

A token can remain watch-only if:

* attention is unclear but structure is strong
* concentration is elevated but no exit behavior exists
* migration behavior is uncertain but not toxic
* state change exists with subcritical confidence

## 10.4 Decision Posture

Forensics does **not** authorize execution. It emits:

* clean
* watch
* toxic
* invalid
* post_rug
* monitor_for_reversal

* **Low Cap Hunter** decides what is early and exploitable
* **Shadow Intelligence** detects when tracked tokens materially change state
* **Forensics** feeds all three, but authorizes none of them
* **Deterministic Core** remains the only authority shell

# 17. Closing Instruction

## Required closing instruction for implementation use

Any implementation based on this spec must follow this order:

1. **Do not redesign authority boundaries**
2. **Wire the missing analyzers first**
3. **Normalize outputs into one shared forensic contract**
4. **Route those outputs explicitly into Meta Fetch, Low Cap Hunter, and Shadow Intelligence**
5. **Package replay evidence for flagged tokens**
6. **Keep all outputs non-authoritative outside the deterministic core**
7. **Reject any implementation that widens MCP or sidecar authority**
8. **Reject any implementation that creates a second decision-truth path**
9. **Prefer fail-closed behavior when forensic inputs are missing, invalid, or contradictory**
10. **Treat this document as an architecture-and-logic-flow contract, not as optional guidance**

## Final rule

If there is a conflict between convenience and architecture truth:

**architecture truth wins**.
